## <a id="структура-проекта"></a>2. Структура проекта

Проект организован как Gradle-мультипроект, где каждый модуль отвечает за свой уровень тестирования или функциональности.

```text
.
├── target-app/                  # Целевое Spring Boot приложение (мишень)
│   ├── src/main/kotlin/         # Бизнес-логика, контроллеры, конфигурация Resilience4j
│   └── src/main/resources/      # Конфигурация приложения и SQL-миграции Flyway
├── performance-tests/           # Модули тестирования производительности
│   ├── benchmarks/              # JMH микробенчмарки (изолированные тесты производительности кода)
│   ├── load-tests/              # Gatling сценарии (профили нагрузки: smoke, load, stress и др.)
│   ├── chaos-tests/             # Тесты устойчивости (Toxiproxy + Chaos Monkey)
│   └── infra/                   # Инфраструктурное окружение (Docker Compose, Prometheus, Grafana)
├── reports/                     # Агрегированные отчёты всех инструментов
│   ├── gatling/                 # Интерактивные HTML-отчёты Gatling
│   └── jmh_results.json         # Результаты последнего прогона JMH для сравнения с baseline
├── build.gradle.kts             # Корневой скрипт сборки, управление общими зависимостями и плагинами
└── settings.gradle.kts          # Определение структуры мультипроекта
```

## 3. Целевое приложение (target-app)

Приложение на **Spring Boot 3.4** и **Kotlin** специально спроектировано как мишень для всех трёх уровней тестирования. Оно покрывает типичные узкие места реальных микросервисов: сериализацию, работу с кэшем, запросы к БД, HTTP-вызовы и CPU-нагрузку.

### Используемые библиотеки

| Библиотека | Версия | Назначение |
|---|---|---|
| **Spring Boot** | 3.4.3 | Основной фреймворк, IoC, REST |
| **Kotlin** | 2.0.0 | Язык реализации |
| **Jackson** | (BOM) | JSON-сериализация, основная |
| **kotlinx.serialization** | 1.6.3 | JSON-сериализация, альтернативная |
| **Gson** | 2.10.1 | JSON-сериализация, для сравнения |
| **OkHttpClient** | 4.12.0 | HTTP-клиент с пулом соединений |
| **Caffeine** | 3.1.8 | In-process LRU-кэш |
| **Spring Data Redis** | (BOM) | Distributed-кэш через Lettuce |
| **Flyway** | (BOM) | Версионирование схемы БД |
| **PostgreSQL JDBC** | (BOM) | Драйвер базы данных |
| **Resilience4j** | 2.2.0 | Circuit Breaker, Retry |
| **Chaos Monkey** | 3.1.0 | Инъекция сбоев через Actuator |
| **Micrometer + Prometheus** | (BOM) | Экспорт метрик |

### REST API

| Эндпоинт | Метод | Назначение |
|---|---|---|
| `/api/ping` | GET / POST | Healthcheck, симуляция Auth |
| `/api/serialize/jackson` | POST | Сериализация через Jackson |
| `/api/serialize/kotlinx` | POST | Сериализация через kotlinx.serialization |
| `/api/serialize/gson` | POST | Сериализация через Gson |
| `/api/db/query?id=` | GET | Запрос к PostgreSQL с Circuit Breaker |
| `/api/cache/caffeine?key=` | GET | Чтение из Caffeine |
| `/api/cache/redis?key=` | GET | Чтение из Redis |
| `/api/http/okhttp` | GET | HTTP-запрос через OkHttpClient |
| `/api/http/webclient` | GET | HTTP-запрос через OkHttpClient |
| `/api/algo/cpu?iterations=` | GET | CPU-задача (подсчёт простых чисел) |

### Resilience4j Circuit Breaker

Защищает два критичных направления — запросы к БД и внешние HTTP-вызовы. При 50% ошибок за скользящее окно из 10 запросов Circuit Breaker переходит в состояние OPEN и автоматически вызывает fallback-методы, возвращающие деградированный, но корректный ответ.

## 4. Управление схемой БД — Flyway

Схема базы данных управляется через **Flyway** — инструмент версионирования SQL-миграций. При каждом старте приложения Flyway автоматически проверяет и применяет все новые миграции, гарантируя консистентность схемы между окружениями.

Миграции хранятся в `src/main/resources/db/migration/` и именуются по соглашению `V{версия}__{описание}.sql`:

```
db/migration/
└── V1__create_items_table.sql
```

Добавление новых изменений схемы — это просто новый SQL-файл с увеличенным номером версии. Flyway хранит историю применённых миграций в таблице `flyway_schema_history`.

## 5. HTTP-клиент — OkHttpClient

Все HTTP-вызовы выполняются через **OkHttpClient** — зрелую и высокопроизводительную библиотеку с:

- **Встроенным пулом соединений** — переиспользование keep-alive соединений снижает латентность
- **Прозрачным retry** при сбоях соединения
- **Настраиваемыми таймаутами** на уровне connect и read
- **Поддержкой HTTP/2** из коробки

Единственный экземпляр `OkHttpClient` создаётся при старте приложения и переиспользуется во всех запросах — именно так рекомендует использовать его документация Square.

## 6. Сериализация

Приложение реализует и сравнивает три подхода к JSON-сериализации:

| Библиотека | Подход | Особенности |
|---|---|---|
| **Jackson** | Рефлексия + кэш метаданных | Наиболее гибкий, богатая экосистема аннотаций |
| **Gson** | Рефлексия | Простой API, минимум зависимостей |
| **kotlinx.serialization** | Кодогенерация (KSP) | Нулевая рефлексия в рантайме, идиоматичен для Kotlin |

`ObjectMapper` (Jackson) и `Json` (kotlinx) создаются по одному экземпляру на всё приложение — оба класса потокобезопасны и спроектированы для многократного переиспользования.

## 7. JMH-микробенчмарки

**JMH (Java Microbenchmark Harness)** — стандарт де-факто для микробенчмарков на JVM. Он корректно обрабатывает прогрев JIT-компилятора, dead code elimination и другие артефакты, которые делают наивные бенчмарки ненадёжными.

### Бенчмарки

| Класс | Режим | Что измеряется |
|---|---|---|
| `SerializationBenchmark` | Throughput (ops/s) | Jackson vs Gson vs kotlinx при разном размере payload |
| `CacheBenchmark` | AverageTime (ms) | Caffeine: Cache Hit vs гарантированный Cache Miss |
| `HttpClientBenchmark` | Throughput (ops/s) | OkHttpClient через MockWebServer |
| `BusinessLogicBenchmark` | Throughput (ops/s) | Сортировка и бинарный поиск |

### CI-регрессионная проверка

`BenchmarkRunner` автоматически сравнивает результаты с baseline. При первом запуске создаётся `jmh_baseline.json`. В последующих запусках деградация **>10%** от baseline вызывает `RuntimeException` и блокирует сборку.

```
Первый запуск  →  jmh_results.json  →  копируется в jmh_baseline.json
Следующие      →  jmh_results.json  →  сравнение с baseline
                                              │
                                    деградация ≤ 10% → ✅ OK
                                    деградация > 10% → ❌ сборка падает
```

## 8. Нагрузочные тесты — Gatling

**Gatling** — высокопроизводительный инструмент нагрузочного тестирования на Scala/Kotlin с реактивной архитектурой. Один процесс Gatling способен эмулировать тысячи виртуальных пользователей без накладных расходов на потоки.

### Профили нагрузки

| Профиль | Нагрузка | Длительность | Цель |
|---|---|---|---|
| `smoke` | 5 VU | ~30 сек | Базовая проверка работоспособности |
| `load` | 100 → 500 RPS | 10 мин | Рабочая нагрузка |
| `stress` | → 2000 RPS | 5 мин | Поиск предела производительности |
| `soak` | 200 RPS | 60 мин | Долгосрочная стабильность, утечки памяти |
| `spike` | 10 → 1000 → 10 RPS | ~50 сек | Поведение при резких пиках |

### Стратегия фидеров

Для профилей с высоким RPS используется динамический UUID-фидер — каждая виртуальная сессия получает уникальный ключ, что исключает аномальное попадание в кэш и даёт честную картину производительности Redis, Caffeine и БД под нагрузкой.

## 9. Chaos Engineering

**Chaos Engineering** — практика намеренного внесения сбоев для проверки устойчивости системы. Используются два инструмента:

### Toxiproxy

Прокси-сервер для симуляции сетевых условий между сервисами:

| Сценарий | Toxic | Что проверяет |
|---|---|---|
| DB Chaos | `latency` (500ms + 100ms jitter) | Срабатывание Circuit Breaker на задержках БД |
| Cache Chaos | `bandwidth=0` (полная блокировка) | Автоматический fallback при недоступности Redis |

### Chaos Monkey for Spring Boot

Библиотека для инъекции сбоев непосредственно в Spring-приложение через Actuator:

| Сценарий | Assault | Что проверяет |
|---|---|---|
| Downstream Chaos | `latencyActive` | Устойчивость при задержках на HTTP-вызовах |
| Memory Pressure | `memoryActive` | Поведение GC и health при атаке на Heap |
| Pod Kill | shutdown endpoint | Graceful shutdown приложения |

Если Toxiproxy недоступен, тесты автоматически переходят в mock-режим и проверяют корректность fallback-поведения напрямую.

> ⚠️ В текущей версии `target-app` подключается к Postgres/Redis напрямую, минуя прокси-порты Toxiproxy, поэтому таблица выше описывает *задуманное* поведение, а не гарантированно наблюдаемое из коробки — см. «Известные ограничения» в README.

## 10. Мониторинг и наблюдаемость

### Метрики приложения

**Micrometer** собирает метрики JVM и приложения и экспортирует их в формате Prometheus через `/actuator/prometheus`. Prometheus scrape-ит эндпоинт по расписанию, Grafana строит дашборды.

Ключевые метрики:
- `http_server_requests_seconds` — латентность и RPS по каждому эндпоинту
- `resilience4j_circuitbreaker_state` — текущее состояние Circuit Breaker
- `jvm_memory_used_bytes` — использование памяти JVM (heap + non-heap)
- `hikaricp_connections_active` — активные соединения пула HikariCP

### Метрики нагрузочных тестов

Gatling по умолчанию пишет интерактивные HTML-отчёты локально в `reports/gatling/`. Контейнер **InfluxDB** в `docker-compose.yml` поднят как заготовка инфраструктуры для live-метрик, но интеграция не завершена: для реальной записи в реальном времени нужно добавить `gatling.conf` с настройкой InfluxDB/Graphite data writer. Без этого шага датасорс InfluxDB в Grafana будет пустым, а дашборд Gatling в Grafana показывать нечего — это известное ограничение текущей версии (см. README → «Известные ограничения»).
